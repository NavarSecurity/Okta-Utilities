# okta-app-provisioning-exporter

Python utility for exporting Okta application provisioning configuration, app user schemas, profile mappings, and connector metadata.

This utility is intended for Okta migration support, application onboarding review, provisioning troubleshooting, SCIM readiness analysis, connector inventory, and implementation evidence collection.

---

## What This Utility Does

The `okta-app-provisioning-exporter` utility exports provisioning-related application configuration from an Okta org.

It can export:

| Export Area | Purpose |
|---|---|
| App inventory | Lists selected Okta applications and high-level app metadata. |
| Provisioning indicators | Captures provisioning feature flags and settings visible from the Okta app object. |
| App user schemas | Exports app profile schema attributes where the app exposes a schema. |
| Profile mappings | Exports mappings where the selected app participates as source or target. |
| Connector metadata | Summarizes connector name, label, sign-on mode, status, features, and available links. |
| Raw redacted app objects | Preserves app configuration for review while redacting secrets. |

This utility does not create, update, delete, assign, or provision users. It is read-only.

---

## Safety Model

This utility is read-only.

It does not modify Okta apps, schemas, mappings, users, groups, assignments, or provisioning settings.

Dry-run validation:

```bash
okta-app-provisioning-exporter --config config.json --dry-run
```

Real export:

```bash
okta-app-provisioning-exporter --config config.json
```

The utility should not print or write Okta API tokens, client secrets, private keys, provisioning credentials, bearer tokens, authorization headers, or password values.

## Requirements

- Python 3.10 or newer
- Okta admin API token
- Okta org URL
- Network access to the Okta tenant

The API token should be generated by an Okta admin account with permission to read applications, schemas, and profile mappings.

## Setup

Run these commands from the utility folder.

Create a virtual environment:

```bash
python3 -m venv .venv
```

Activate the virtual environment on macOS or Linux:

```bash
source .venv/bin/activate
```


```bash
python -m pip install -e .
```

```bash
cp .env.example .env
```

## Configuration

The utility is driven by `config.json`.

Example full export configuration:

```json
{
  "outputDirectory": "output",
  "includeInactiveApps": false,
  "includeAppSchemas": true,
  "includeProfileMappings": true,
  "includeAppFeatures": true,
  "includeConnectorDetails": true,
  "skipOktaSystemApps": true,
  "continueOnAppSchemaError": true,
  "redactSensitiveValues": true,
  "appSelection": {
    "mode": "all",
    "appIds": [],
    "appNames": [],
    "appFile": "input/apps.txt"
  },
  "excludedAppNames": [
    "saasure",
    "okta_enduser",
    "okta_browser_plugin",
    "okta_oin_submission_tester_app",
    "okta_iga_reviewer"
  ],
  "timeoutSeconds": 30
}
```

Example selected app configuration:

```json
{
  "outputDirectory": "output",
  "includeInactiveApps": false,
  "includeAppSchemas": true,
  "includeProfileMappings": true,
  "includeAppFeatures": true,
  "includeConnectorDetails": true,
  "skipOktaSystemApps": true,
  "continueOnAppSchemaError": true,
  "redactSensitiveValues": true,
  "appSelection": {
    "mode": "file",
    "appIds": [],
    "appNames": [],
    "appFile": "input/apps.txt"
  },
  "timeoutSeconds": 30
}
```

Supported `appSelection.mode` values:

| Mode | Behavior |
|---|---|
| `all` | Export all eligible apps. |
| `ids` | Export apps whose IDs are listed in `appIds`. |
| `names` | Export apps whose names or labels are listed in `appNames`. |
| `file` | Export apps listed in `appFile`. |

The app file can contain app IDs, app internal names, or app labels, one per line.

## What the Output Files Mean

| File | Purpose |
|---|---|
| `apps_full.json` | Redacted full Okta app objects for selected apps. |
| `apps_summary.csv` | Flat inventory of selected apps. |
| `connector_details.csv` | Connector-oriented app summary including app name, label, sign-on mode, status, and link metadata. |
| `provisioning_features.csv` | Provisioning-related app features such as push, import, profile mastering, group push, and schema import flags. |
| `app_schemas_full.json` | Full app user profile schemas where available. |
| `app_schema_attributes.csv` | Flat list of app schema attributes. |
| `profile_mappings_full.json` | Profile mappings related to selected apps. |
| `profile_mapping_properties.csv` | Attribute-level mapping details for selected app mappings. |
| `selected_apps.json` | Apps included in the export. |
| `skipped_apps.json` | Apps skipped because they were inactive, system apps, or not selected. |
| `skipped_apps.csv` | Flat review file for skipped apps. |
| `app_schema_failures.csv` | Apps where schema export failed but the run continued. |
| `feature_request_failures.csv` | Apps where feature export failed for reasons other than known non-applicable provisioning support. |
| `feature_not_applicable.csv` | Apps where Okta reported that provisioning features are not supported. These are informational, not request failures. |
| `execution_report.json` | Run status, counts, warnings, and failures. |
| `manifest.json` | Run metadata and output file inventory. |
| `dry_run_report.json` | Dry-run validation result and planned export outputs. Created only during dry run. |
| `config_summary.json` | Non-secret summary of selected configuration. Created during dry run. |

## How to Read Provisioning Features

Some Okta apps expose provisioning features such as:

```text
PUSH_NEW_USERS
PUSH_PROFILE_UPDATES
PUSH_PASSWORD_UPDATES
IMPORT_NEW_USERS
IMPORT_PROFILE_UPDATES
IMPORT_USER_SCHEMA
PROFILE_MASTERING
GROUP_PUSH
```

Read these as indicators of what the app connector can support or currently has available in the exported app object or feature endpoint. Availability can vary by app integration, Okta org, connector type, and licensing.

A useful review pattern:

```text
PUSH_NEW_USERS present          = Okta may be able to create users in the target app.
PUSH_PROFILE_UPDATES present    = Okta may be able to update app user profiles.
IMPORT_NEW_USERS present        = Okta may be able to import users from the app.
PROFILE_MASTERING present       = The app may be configured or capable as a profile source.
GROUP_PUSH present              = Okta may be able to push groups to the downstream app.
```

Always validate the actual app provisioning settings in the Okta Admin Console before making production decisions.

## Troubleshooting

### 401 Unauthorized

Confirm the Okta org URL and API token in `.env`.

```bash
cat .env
```

The org URL should look like:

```text
https://your-org.okta.com
```

Do not include `/api/v1` in `OKTA_ORG_URL`.

---

### 403 Forbidden

The API token may not have enough permission to read applications, schemas, or profile mappings.

Use an admin-generated API token with the required read permissions.

---

### App schema failures

Some Okta apps do not expose an app user profile schema through the schema endpoint.

If `continueOnAppSchemaError` is true, the utility records the failure in `app_schema_failures.csv` and continues.

To reduce noise, keep this enabled:

```json
{
  "skipOktaSystemApps": true
}
```

---

### Feature not applicable

Some apps do not support provisioning features. When Okta returns a response such as `Provisioning is not supported.`, the utility writes the app to `feature_not_applicable.csv` and does not count it as a request failure.

This is expected for apps that can exist in Okta but do not support provisioning feature metadata.

### Feature request failures

If the feature request fails for another reason, the utility writes the failure to `feature_request_failures.csv`, records a warning, and continues.

Use `feature_request_failures.csv` to review unexpected API failures. Use `feature_not_applicable.csv` to review apps where provisioning features are simply not available.

---

### Too many apps exported

Use selected app mode.

```json
{
  "appSelection": {
    "mode": "file",
    "appFile": "input/apps.txt"
  }
}
```

Then list only the apps you want in `input/apps.txt`.

## Security Notes

- Do not commit `.env`.
- Do not commit raw client exports unless they have been reviewed.
- Treat app provisioning configuration as sensitive because it may reveal connector design, app names, app URLs, profile flows, and downstream system details.
- Review exported app settings before sharing them outside the project team.
- Keep output reports with the project evidence package when used during a migration or audit review.

---

## Practical Use Cases

Use this utility to:

- Inventory apps with provisioning-related features.
- Review app user schemas before migration.
- Export connector metadata before recreating apps in another org.
- Identify apps that depend on profile mappings.
- Compare provisioning design across projects or environments.
- Support SCIM and app provisioning troubleshooting.
- Build evidence for Okta app onboarding or migration work.

---

## Notes

This utility exports Okta-side application provisioning configuration and related metadata. It does not connect to downstream SaaS apps, does not test SCIM endpoints, and does not validate whether provisioning succeeds in the target application.
