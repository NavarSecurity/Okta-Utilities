# okta-profile-mapping-exporter

Python utility for exporting Okta profile mappings between Okta user profiles, app user profiles, directory profiles, and external identity provider profiles.

This utility is intended for Okta migration support, Universal Directory reviews, app onboarding analysis, profile mapping cleanup, and implementation evidence collection.

---

## What This Utility Does

The `okta-profile-mapping-exporter` utility exports Okta profile mapping configuration from an Okta org.

It can export:

| Export Area | Purpose |
|---|---|
| Profile mapping inventory | Lists profile mappings configured in the org. |
| Mapping details | Captures source, target, direction, and mapping metadata. |
| Attribute-level mappings | Flattens profile mapping properties into reviewable CSV output. |
| Source and target summaries | Shows what systems participate in mappings. |
| Mapping JSON | Preserves the full raw API response for backup and deeper review. |

Profile mappings define how attributes move between profiles, such as Okta user profile to app user profile, app user profile back to Okta, directory profile to Okta, or IdP profile to Okta.

---

## Safety Model

This utility is read-only.

It does not create, update, delete, or modify profile mappings.

Dry-run validation:

```bash
okta-profile-mapping-exporter --config config.json --dry-run
```

Real export:

```bash
okta-profile-mapping-exporter --config config.json
```

The utility should not print or write Okta API tokens, client secrets, private keys, or authorization headers.

## Requirements

- Python 3.10 or newer
- Okta admin API token
- Okta org URL
- Network access to the Okta tenant

The API token should be generated by an Okta admin account with permission to read profile mappings.

## Setup

Run these commands from the utility folder.

Create a virtual environment:

```bash
python3 -m venv .venv
```

Activate the virtual environment on macOS or Linux:

```bash
source .venv/bin/activate
```

Activate the virtual environment on Windows PowerShell:

```powershell
.venv\Scripts\Activate.ps1
```

Install the utility:

```bash
python -m pip install -e .
```

Copy the environment example:

```bash
cp .env.example .env
```

Copy a sample configuration file:

```bash
cp samples/config.export.sample.json config.json
```

Open `.env` and update the Okta values.

## Configuration

The utility is driven by `config.json`.

Example default export configuration:

```json
{
  "outputDirectory": "output",
  "limit": 200,
  "includeMappingDetails": true,
  "filters": {
    "sourceId": "",
    "targetId": ""
  },
  "redactSensitiveValues": true,
  "timeoutSeconds": 30,
  "retry": {
    "maxAttempts": 3,
    "backoffSeconds": 1
  }
}
```

Example filtered export configuration:

```json
{
  "outputDirectory": "output",
  "limit": 200,
  "includeMappingDetails": true,
  "filters": {
    "sourceId": "replace-with-source-profile-id",
    "targetId": ""
  },
  "redactSensitiveValues": true,
  "timeoutSeconds": 30,
  "retry": {
    "maxAttempts": 3,
    "backoffSeconds": 1
  }
}
```

Use `sourceId` or `targetId` only when you want to limit the export to mappings for a specific profile source or target.

## Operation: Dry Run

Use dry run to validate configuration and environment setup before making live API calls.

```bash
okta-profile-mapping-exporter --config config.json --dry-run
```

Expected output:

```text
output/profile-mapping-export-dry-run-<timestamp>/
  execution_report.json
  manifest.json
```

Dry run does not call Okta.

---

## Operation: Export Profile Mappings

Use export mode to capture the current Okta profile mapping configuration.

```bash
okta-profile-mapping-exporter --config config.json
```

Expected output:

```text
output/profile-mapping-export-<timestamp>/
  mappings_full.json
  profile_mappings_summary.csv
  profile_mapping_properties.csv
  mapping_sources_targets.csv
  execution_report.json
  manifest.json
  mappings_by_source_type/
  mappings_by_target_type/
```

Use this before migration work, app onboarding, Universal Directory cleanup, or profile mapping changes.

---

## Output Reports

Each run creates timestamped output.

| File | Purpose |
|---|---|
| `mappings_full.json` | Full profile mapping export from Okta. |
| `profile_mappings_summary.csv` | One row per profile mapping. |
| `profile_mapping_properties.csv` | One row per mapped target attribute. |
| `mapping_sources_targets.csv` | Distinct source and target profile objects used by mappings. |
| `execution_report.json` | Counts, warnings, errors, and run metadata. |
| `manifest.json` | Input files, output files, operation, and timestamp. |
| `mappings_by_source_type/` | JSON files grouped by mapping source type. |
| `mappings_by_target_type/` | JSON files grouped by mapping target type. |

Execution reports should not include API tokens or sensitive headers.

## Recommended Workflow

Export profile schemas first with utility 36:

```bash
okta-profile-schema-exporter --config config.json
```

Then export profile mappings with this utility:

```bash
cp samples/config.export.sample.json config.json
okta-profile-mapping-exporter --config config.json
```

Review the flat mapping output:

```text
output/profile-mapping-export-<timestamp>/profile_mapping_properties.csv
```

Use the CSV to answer questions such as:

```text
Which attributes are mapped from Okta to each app?
Which apps depend on a custom Okta user attribute?
Which mappings use Okta Expression Language?
Which mappings push data from an app, directory, or IdP into Okta?
Which required app attributes are missing mappings?
```

## Troubleshooting

### 401 Unauthorized

Confirm the Okta org URL and API token in `.env`.

```bash
cat .env
```

The org URL should look like:

```text
https://your-org.okta.com
```

Do not include `/api/v1` in `OKTA_ORG_URL`.

---

### 403 Forbidden

The API token may not have permission to read profile mappings.

Confirm the token was generated by an admin account with sufficient permissions.

---

### No mappings exported

Confirm the org has profile mappings configured.

Also check whether `sourceId` or `targetId` filters are limiting the results.

Use the default sample config for a broad export:

```bash
cp samples/config.export.sample.json config.json
okta-profile-mapping-exporter --config config.json
```

---

### Attribute mappings are hard to read

Open the flat CSV output:

```text
output/profile-mapping-export-<timestamp>/profile_mapping_properties.csv
```

This file is easier to review than the full JSON because each target attribute mapping is shown as a separate row.

## Security Notes

- Do not commit `.env`.
- Do not commit raw client exports unless they have been reviewed.
- Treat profile mappings as sensitive because they may expose HR attributes, directory attributes, app attributes, and Okta Expression Language logic.
- Review generated exports before sharing them with clients or vendors.
- Keep execution reports with the project evidence package.

---

## Practical Use Cases

Use this utility to:

- Export all Universal Directory profile mappings before a migration.
- Review which Okta attributes are mapped to apps.
- Identify mappings that use custom Okta user attributes.
- Compare mapping inventory before and after app onboarding.
- Prepare evidence for IAM discovery or implementation handoff.
- Support profile schema cleanup and attribute rationalization.
- Feed future mapping comparison or migration utilities.

---

## Notes

Profile mappings can directly affect provisioning behavior, app profile data, and inbound identity data.

For production environments, export profile schemas with utility 36 and profile mappings with utility 38 before making Universal Directory or app provisioning changes.
