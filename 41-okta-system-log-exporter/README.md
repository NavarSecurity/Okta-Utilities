# okta-system-log-exporter

Python utility for exporting Okta System Log events by time range, event type, actor, target, outcome, query text, or custom System Log filter.

This utility is intended for Okta audit support, incident review, troubleshooting, operational evidence collection, security investigations, and post-change validation.

---

## What This Utility Does

The `okta-system-log-exporter` utility exports System Log events from an Okta org and writes structured reports for review.

It can export:

| Export Area | Purpose |
|---|---|
| Raw System Log events | Preserves full redacted event objects returned by Okta. |
| Flat event CSV | Provides a readable event-level CSV for audit, investigation, and handoff. |
| Event type summary | Counts events by `eventType`. |
| Actor summary | Counts activity by actor ID, login, display name, and actor type. |
| Target summary | Counts target references across exported events. |
| Outcome summary | Counts results such as `SUCCESS`, `FAILURE`, `ALLOW`, or `DENY` where present. |
| Events by type | Writes grouped JSON files by event type for easier review. |

This utility does not create, update, delete, assign, or modify anything in Okta. It is read-only.

---

## Safety Model

This utility is read-only.

It only calls the Okta System Log API.

Dry-run validation:

```bash
okta-system-log-exporter --config config.json --dry-run
```

Real export:

```bash
okta-system-log-exporter --config config.json
```

The utility should not print or write Okta API tokens, bearer tokens, authorization headers, cookies, passwords, client secrets, private keys, or other secret values.

## Filter Behavior

The utility can use either a custom Okta System Log filter or generate a filter from the config fields.

Use `filter` when you already know the exact Okta filter expression:

```json
{
  "filter": "eventType eq \"user.authentication.failed\" and outcome.result eq \"FAILURE\""
}
```

Use structured fields when you want the utility to build the filter:

```json
{
  "eventTypes": [
    "user.authentication.failed"
  ],
  "outcomes": [
    "FAILURE"
  ],
  "actorLogins": [
    "admin@example.com"
  ]
}
```

If both a custom `filter` and structured filter fields are provided, the utility combines them with `and`.

## Operation: Export System Log Events

Run a real export:

```bash
okta-system-log-exporter --config config.json
```

Expected output:

```text
output/system-log-export-<timestamp>/
  system_log_events_full.json
  system_log_events.csv
  event_type_summary.csv
  actor_summary.csv
  target_summary.csv
  outcome_summary.csv
  execution_report.json
  manifest.json
  events_by_type/
```

The utility follows Okta pagination links until there are no more events, the configured `maxEvents` limit is reached, or an error occurs.

## Output Reports

Each run creates timestamped output.

Common files:

| File | Purpose |
|---|---|
| `system_log_events_full.json` | Full redacted System Log event export. |
| `system_log_events.csv` | Flat event-level CSV for review. |
| `event_type_summary.csv` | Event counts by event type. |
| `actor_summary.csv` | Event counts grouped by actor. |
| `target_summary.csv` | Event counts grouped by target. |
| `outcome_summary.csv` | Event counts grouped by outcome result. |
| `events_by_type/` | Per-event-type JSON files. |
| `execution_report.json` | Counts, warnings, errors, and run metadata. |
| `manifest.json` | Input files, output files, operation, and timestamp. |

Execution reports should not include API tokens or sensitive headers.

## Troubleshooting

### 401 Unauthorized

Confirm the Okta org URL and API token in `.env`.

```bash
cat .env
```

The org URL should look like:

```text
https://your-org.okta.com
```

Do not include `/api/v1` in `OKTA_ORG_URL`.

---

### 403 Forbidden

The API token may not have enough permission to read System Log events.

Confirm the token was generated by an Okta admin account with the required permissions.

---

### 400 Bad Request

The custom `filter` value may be invalid.

First test with a simple config such as:

```json
{
  "relativeHours": 24,
  "filter": null,
  "eventTypes": [],
  "outcomes": []
}
```

Then add filters one at a time.

---

### No events exported

Possible causes:

```text
The time range is too narrow.
The filter is too restrictive.
The event type does not exist in the selected time range.
The actor or target value is incorrect.
```

Try increasing `relativeHours` or removing filters.

---

### Rate limit responses

The utility attempts to handle `429` responses using the Okta rate-limit reset header where available.

If rate limits continue, reduce `limit`, reduce the time range, or run the export later.

## Security Notes

- Do not commit `.env`.
- Treat System Log exports as sensitive security data.
- Review exports before placing them in shared repositories or project workspaces.
- System Log events can reveal user activity, admin actions, app names, IP addresses, device context, and authentication behavior.
- Use narrower filters when collecting evidence for client handoff.
- Store investigation outputs in an approved project evidence location.

---

## Practical Use Cases

Use this utility to:

- Export login failures for an incident review.
- Pull SSO events for a specific app.
- Review admin activity in a defined time window.
- Collect authentication events before or after a policy change.
- Export rate-limit warnings or suspicious activity indicators.
- Build evidence packages for audit, troubleshooting, or migration validation.
- Feed utility 42 security event detection with System Log exports.

---

## Notes

Okta System Log data is security-sensitive. Keep exports scoped, reviewed, and stored carefully.

For long-running exports, prefer following the utility's pagination behavior rather than manually stitching together separate time windows.
