# okta-api-access-auditor

Python utility for auditing Okta API tokens, OAuth service apps, OAuth scope grants, client admin role assignments, and overprivileged API access patterns.

This utility is intended for Okta API access review, non-human identity review, least-privilege assessment, audit preparation, migration cleanup, and operational evidence collection.

---

## What This Utility Does

The `okta-api-access-auditor` utility exports and reviews Okta API access objects from an org.

It can export:

| Export Area | Purpose |
|---|---|
| API token metadata | Identifies active SSWS API tokens, ownership metadata, token age, last-used data, expiration data, and network restriction metadata where available. |
| OAuth service apps | Identifies OIDC/API service apps that can request machine-to-machine access. |
| OAuth app grants | Exports Okta API OAuth scope grants assigned to selected apps. |
| Client admin roles | Exports admin roles assigned to OAuth client/service apps. |
| High-risk clients | Flags OAuth clients with high-risk scopes, high-risk admin roles, or inconsistent scope/role posture. |
| Risk findings | Produces a flat finding report for API tokens and OAuth clients that need review. |
| Request failures | Records endpoint failures without exposing secrets. |

This utility does not create, update, revoke, rotate, delete, assign, or remove API tokens, apps, scopes, or admin roles. It is read-only.

---

## Safety Model

This utility is read-only.

It does not modify Okta API tokens, OAuth clients, app grants, admin roles, users, groups, apps, authorization servers, or policies.

Dry-run validation:

```bash
okta-api-access-auditor --config config.json --dry-run
```

Real export:

```bash
okta-api-access-auditor --config config.json
```

The utility should not print or write Okta API tokens, bearer tokens, client secrets, private keys, passwords, authorization headers, or session values.

## Requirements

- Python 3.10 or newer
- Okta admin API token
- Okta org URL
- Network access to the Okta tenant

The API token should be generated by an Okta admin account with permission to read API tokens, apps, app grants, and client role assignments.

For OAuth-scoped access, equivalent read scopes can include `okta.apiTokens.read`, `okta.apps.read`, `okta.appGrants.read`, and `okta.roles.read`, depending on the exact endpoints and org configuration. This utility uses SSWS token authentication by default.

```json
  "includeClientRoleAssignments": true,
  "includeInactiveApps": false,

  "redactSensitiveValues": true,

    "mode": "service",

  "riskRules": {
    "staleApiTokenDays": 90,
    "oldApiTokenDays": 365,
    "staleAppDays": 180,
    "highRiskScopes": [
      "okta.apiTokens.manage",
      "okta.apps.manage",
      "okta.authorizationServers.manage",
      "okta.clients.manage",
      "okta.groups.manage",
      "okta.policies.manage",
      "okta.roles.manage",
      "okta.users.manage"

```

## App Selection Modes

Use `appSelection.mode` to control which OAuth apps are audited.

| Mode | Behavior |
|---|---|
| `service` | Selects OIDC apps that appear to be API service/client-credentials apps. This is the default. |
| `all_oidc` | Selects all OIDC apps. |
| `all` | Selects all apps returned by the Apps API. Use carefully. |
| `ids` | Selects apps listed in `appIds`. |
| `names` | Selects apps listed in `appNames`. |
| `file` | Selects apps listed in `input/apps.txt`. |

For selected app mode, copy the sample app list:

```bash
cp samples/apps.txt input/apps.txt
```

Open `input/apps.txt` and list Okta app IDs or app labels, one per line.

## How to Read the Main Reports

### `api_access_risk_findings.csv`

Start here for review.

This file contains one row per finding.

Important columns:

| Column | Meaning |
|---|---|
| `severity` | Finding severity: `CRITICAL`, `HIGH`, `MEDIUM`, or `LOW`. |
| `findingCode` | Short identifier for the issue. |
| `objectType` | Whether the issue is tied to an API token or OAuth client. |
| `objectId` | Token ID or client ID. |
| `objectName` | Token name or app label. |
| `detail` | Human-readable explanation of the issue. |

Example interpretation:

```text
HIGH_RISK_OAUTH_SCOPE means the OAuth client has a granted Okta API scope that can perform privileged management actions.
```

### `high_risk_clients.csv`

This file summarizes OAuth clients with notable API access risk.

Use it to quickly identify service apps that combine high-risk scopes with high-risk admin roles.

### `api_tokens_summary.csv`

This file summarizes Okta SSWS API token metadata.

Review old tokens, tokens with unknown last-used timestamps, stale tokens, and tokens with broad network restrictions.

### `app_grants.csv`

This file lists Okta API OAuth scopes granted to selected apps.

Example interpretation:

```text
A service app with okta.users.manage can manage user resources if its admin role assignment also permits that action.
```

### `client_role_assignments.csv`

This file lists admin roles assigned to OAuth client apps.

Example interpretation:

```text
A client with SUPER_ADMIN has extremely broad administrative authority and should be reviewed immediately.
```

## Risk Finding Rules

The default rules flag:

```text
Stale API tokens
Old API tokens
API tokens with broad or unknown network restrictions
OAuth clients with high-risk Okta API scopes
OAuth clients with high-risk admin roles
Clients with scopes but no exported role assignment
Clients with role assignments but no exported scope grants
OAuth client apps that have not been updated recently
```

Risk findings are review signals, not automatic proof of a misconfiguration. Confirm business purpose, ownership, last use, and required permissions before remediation.

## Troubleshooting

### 401 Unauthorized

Confirm the Okta org URL and API token in `.env`.

```bash
cat .env
```

The org URL should look like:

```text
https://your-org.okta.com
```

Do not include `/api/v1` in `OKTA_ORG_URL`.

---

### 403 Forbidden

The API token may not have permission to read API tokens, apps, app grants, or client role assignments.

Use an admin-generated token with sufficient read access.

---

### App grant requests fail

Some apps may not support app grants or the token may not have permission to read app grants.

Review:

```text
request_failures.csv
execution_report.json
```

Use `appSelection.mode` set to `service`, `all_oidc`, or `file` to reduce unsupported app checks.

---

### Client role assignment requests fail

Client role assignment APIs depend on the client ID, not the Okta app ID.

Confirm selected apps have a value in:

```text
credentials.oauthClient.client_id
```

---

### Request failures appear but the run succeeds

If `continueOnRequestError` is true, the utility records request failures and continues.

Review:

```text
request_failures.csv
execution_report.json
```

This is useful when some endpoints are unavailable because of licensing, org type, API permissions, app type, or feature availability.

## Security Notes

- Do not commit `.env`.
- Do not commit raw API access exports from client environments unless they have been reviewed.
- Treat API token metadata, OAuth grants, and client role assignments as sensitive security evidence.
- Review `api_access_risk_findings.csv` before sharing any report.
- Prefer scoped OAuth service apps over broad SSWS API tokens where practical.
- Validate service app scopes and admin roles together. A scope alone does not fully describe effective access.
- Store output in an approved evidence location.

---

## Practical Use Cases

Use this utility to:

- Identify active SSWS API tokens.
- Review stale or old API tokens.
- Review API token network restriction posture.
- Identify OAuth service apps with Okta API access.
- Export OAuth scope grants for API service apps.
- Export admin roles assigned to OAuth client apps.
- Find service apps with broad scopes or broad admin roles.
- Prepare access review evidence for SOX, ISO, HIPAA, SOC 2, or internal audit.
- Support least-privilege review of API access.
- Prepare for migration from SSWS API tokens to scoped OAuth service apps.

---

## Notes

Okta API access can be highly privileged. Review generated files before storing or sharing them.

For production environments, run the report from a controlled workstation and store output in an approved evidence location.
